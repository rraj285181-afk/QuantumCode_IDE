1. VHDL Introduction

VHDL stands for VHSIC (Very High Speed Integrated Circuit) Hardware Description Language. It is defined by IEEE and used to design, model, and simulate digital electronic systems — from logic gates to complex microprocessors.

Hardware vs Software: The Switch Analogy

Normal software (Python/Java) is like a cooking recipe — execute line by line. VHDL is like home electrical wiring — flip a switch and the bulb turns on instantly, all wires work in parallel at the same time!

Example: Half Adder & Testbench 
library ieee;
use ieee.std_logic_1164.all;

entity Half_Adder is
    port ( a, b : in std_logic; sum, carry : out std_logic );
end entity Half_Adder;

architecture behavioral of Half_Adder is
begin
    sum   <= a xor b;
    carry <= a and b;
end architecture behavioral;

library ieee; use ieee.std_logic_1164.all;
entity tb_Half_Adder is end entity tb_Half_Adder;
architecture sim of tb_Half_Adder is
    signal a, b, sum, carry : std_logic := '0';
begin
    uut: entity work.Half_Adder port map(a=>a, b=>b, sum=>sum, carry=>carry);
    process begin
        a<='0'; b<='0'; wait for 10 ns;
        a<='0'; b<='1'; wait for 10 ns;
        a<='1'; b<='0'; wait for 10 ns;
        a<='1'; b<='1'; wait for 10 ns;
        wait;
    end process;
end architecture sim;

How this Code Works (Simplified)

  • library ieee — Imports standard toolboxes.
  • entity — Declares input/output pins of the chip.
  • architecture — Defines internal logic wiring.
  • wait for 10 ns — Pauses simulation to capture outputs.

2. Entity & Architecture

Every VHDL design has two main parts: the Entity (the interface) and the Architecture (the behaviour).

TV Remote Analogy

  • Entity = The plastic shell with buttons — defines what pins (ports) are visible to the outside world.
  • Architecture = The circuit board inside — defines what happens when a button is pressed.
Example: 2-to-1 Multiplexer 
library ieee; use ieee.std_logic_1164.all;
entity Mux21 is
    port ( sel, d0, d1 : in std_logic; y : out std_logic );
end entity Mux21;
architecture behavioral of Mux21 is
begin
    y <= (not sel and d0) or (sel and d1);
end architecture behavioral;

library ieee; use ieee.std_logic_1164.all;
entity tb_Mux21 is end entity tb_Mux21;
architecture sim of tb_Mux21 is
    signal sel : std_logic := '0';
    signal d0  : std_logic := '1';
    signal d1  : std_logic := '0';
    signal y   : std_logic;
begin
    uut: entity work.Mux21 port map(sel=>sel,d0=>d0,d1=>d1,y=>y);
    process begin
        sel<='0'; wait for 10 ns;
        sel<='1'; wait for 10 ns;
        wait;
    end process;
end architecture sim;

How this Code Works (Simplified)

  • sel='0' — Output y connects to d0.
  • sel='1' — Output y connects to d1.

3. Signals vs Variables

Two ways to hold data in VHDL — Signals (wires) and Variables (scratchpads).

Conveyor Belt vs Notepad Analogy

  • Signal (<=) — Conveyor belt: packages travel and arrive after a delay (updates at end of simulation delta cycle).
  • Variable (:=) — Worker's notepad: writes instantly (updates immediately, only inside process blocks).
Example: Intermediate Signals 
library ieee; use ieee.std_logic_1164.all;
entity Logic_Circuit is
    port ( a, b, c : in std_logic; out_y : out std_logic );
end entity Logic_Circuit;
architecture behavioral of Logic_Circuit is
    signal wire1, wire2 : std_logic;
begin
    wire1 <= a and b;
    wire2 <= not c;
    out_y <= wire1 or wire2;
end architecture behavioral;

library ieee; use ieee.std_logic_1164.all;
entity tb_Logic_Circuit is end entity tb_Logic_Circuit;
architecture sim of tb_Logic_Circuit is
    signal a, b, c : std_logic := '0';
    signal out_y   : std_logic;
begin
    uut: entity work.Logic_Circuit port map(a=>a,b=>b,c=>c,out_y=>out_y);
    process begin
        a<='0'; b<='0'; c<='0'; wait for 10 ns;
        a<='1'; b<='1'; c<='0'; wait for 10 ns;
        a<='0'; b<='1'; c<='1'; wait for 10 ns;
        wait;
    end process;
end architecture sim;

How this Code Works (Simplified)

  • wire1 <= a and b — Intermediate wire carries AND result.
  • out_y <= wire1 or wire2 — Final OR combining both intermediate wires.

4. VHDL Data Types

VHDL is strongly typed — every signal must have a declared type before use.

Road vs Highway Analogy

  • std_logic — One-lane road (single bit: '0','1','Z','X')
  • std_logic_vector(7 downto 0) — 8-lane highway (8 bits in parallel)
  • integer — A number you can add/subtract (for simulation math)
  • boolean — true/false decision gate
Example: 3-Input OR Gate 
library ieee; use ieee.std_logic_1164.all;
entity Or3_Gate is
    port ( x, y, z : in std_logic; out_f : out std_logic );
end entity Or3_Gate;
architecture behavioral of Or3_Gate is
begin
    out_f <= x or y or z;
end architecture behavioral;

library ieee; use ieee.std_logic_1164.all;
entity tb_Or3 is end entity tb_Or3;
architecture sim of tb_Or3 is
    signal x, y, z : std_logic := '0';
    signal out_f   : std_logic;
begin
    uut: entity work.Or3_Gate port map(x=>x,y=>y,z=>z,out_f=>out_f);
    process begin
        x<='0'; y<='0'; z<='0'; wait for 10 ns;
        x<='1'; y<='0'; z<='0'; wait for 10 ns;
        x<='0'; y<='1'; z<='1'; wait for 10 ns;
        wait;
    end process;
end architecture sim;

How this Code Works (Simplified)

  • out_f <= x or y or z — Output turns HIGH if ANY input is '1'.

5. Concurrent Statements

All statements OUTSIDE process blocks run simultaneously — this is the natural state of hardware.

Water Pipes Analogy

Opening your kitchen tap and bathroom tap simultaneously — both flow at the same time, independent of each other. In VHDL, all concurrent signal assignments execute in parallel regardless of their order in the code.

Example: XNOR Comparator 
library ieee; use ieee.std_logic_1164.all;
entity Comparator is
    port ( a, b : in std_logic; equals : out std_logic );
end entity Comparator;
architecture behavioral of Comparator is
begin
    equals <= not (a xor b);
end architecture behavioral;

library ieee; use ieee.std_logic_1164.all;
entity tb_Comparator is end entity tb_Comparator;
architecture sim of tb_Comparator is
    signal a, b, equals : std_logic := '0';
begin
    uut: entity work.Comparator port map(a=>a,b=>b,equals=>equals);
    process begin
        a<='0'; b<='0'; wait for 10 ns;
        a<='0'; b<='1'; wait for 10 ns;
        a<='1'; b<='1'; wait for 10 ns;
        wait;
    end process;
end architecture sim;

How this Code Works (Simplified)

  • not (a xor b) — XNOR gate: outputs '1' only when both inputs are equal.

6. Process Statements

A process block contains sequential code that executes line-by-line — triggered by changes in the sensitivity list.

Security Guard Analogy

The guard sleeps at the gate. The moment a signal in the sensitivity list changes (like clk or reset), the guard wakes up, executes the checklist top-to-bottom, then goes back to sleep.

Example: Process Stimulus 
library ieee; use ieee.std_logic_1164.all;
entity Process_Demo is
    port ( a, b : in std_logic; y : out std_logic );
end entity Process_Demo;
architecture behavioral of Process_Demo is
begin
    y <= a xor b;
end architecture behavioral;

library ieee; use ieee.std_logic_1164.all;
entity tb_Process_Demo is end entity tb_Process_Demo;
architecture sim of tb_Process_Demo is
    signal a, b, y : std_logic := '0';
begin
    uut: entity work.Process_Demo port map(a=>a,b=>b,y=>y);
    process begin
        a<='0'; b<='0'; wait for 10 ns;
        a<='1'; b<='0'; wait for 10 ns;
        a<='1'; b<='1'; wait for 10 ns;
        wait;
    end process;
end architecture sim;

How this Code Works (Simplified)

  • process begin ... wait; — Runs stimulus sequentially, pausing at each wait statement to let the design settle.

7. Control Structures (if-else / case)

Inside process blocks, use if-else and case to conditionally route signals.

Train Track Switcher Analogy

A track lever routes the train to Track A or Track B based on its position. In hardware, this compiles to a multiplexer circuit.

Example: Mux with if-else 
library ieee; use ieee.std_logic_1164.all;
entity Control_Mux is
    port ( sel, d0, d1 : in std_logic; y : out std_logic );
end entity Control_Mux;
architecture behavioral of Control_Mux is
begin
    y <= (d0 and not sel) or (d1 and sel);
end architecture behavioral;

library ieee; use ieee.std_logic_1164.all;
entity tb_Control_Mux is end entity tb_Control_Mux;
architecture sim of tb_Control_Mux is
    signal sel : std_logic := '0';
    signal d0  : std_logic := '1';
    signal d1  : std_logic := '0';
    signal y   : std_logic;
begin
    uut: entity work.Control_Mux port map(sel=>sel,d0=>d0,d1=>d1,y=>y);
    process begin
        sel<='0'; wait for 10 ns;
        sel<='1'; wait for 10 ns;
        d0<='0'; d1<='1'; sel<='0'; wait for 10 ns;
        sel<='1'; wait for 10 ns;
        wait;
    end process;
end architecture sim;

How this Code Works (Simplified)

  • When sel='0', output is d0. When sel='1', output is d1.

8. Sequential Logic & Registers

Sequential circuits store state — they only update on clock edges.

Camera Shutter Analogy

  • Input d = the scene in front of camera (changes freely)
  • Clock clk = the shutter button
  • Output q = the frozen photo on screen (only updates when shutter is pressed)
Example: D-Latch 
library ieee; use ieee.std_logic_1164.all;
entity DLatch is
    port ( clk, d : in std_logic; q : inout std_logic );
end entity DLatch;
architecture behavioral of DLatch is
begin
    q <= (d and clk) or (q and not clk);
end architecture behavioral;

library ieee; use ieee.std_logic_1164.all;
entity tb_DLatch is end entity tb_DLatch;
architecture sim of tb_DLatch is
    signal clk, d : std_logic := '0';
    signal q : std_logic;
begin
    uut: entity work.DLatch port map(clk=>clk,d=>d,q=>q);
    process begin
        clk<='0'; d<='1'; wait for 10 ns;
        clk<='1'; d<='1'; wait for 10 ns;
        clk<='0'; d<='0'; wait for 10 ns;
        clk<='1'; d<='0'; wait for 10 ns;
        wait;
    end process;
end architecture sim;

How this Code Works (Simplified)

  • When clk='1', q follows d. When clk='0', q holds its last value.

9. Structural Modeling

Build complex circuits by connecting sub-modules together like LEGO bricks.

LEGO Blocks Analogy

Instead of moulding custom plastic, grab standard blocks (AND, OR, NOT gates) and snap them together. Component instantiation = connecting LEGO pieces.

Example: Structural XOR 
library ieee; use ieee.std_logic_1164.all;
entity Structural_Xor is
    port ( a, b : in std_logic; y : out std_logic );
end entity Structural_Xor;
architecture structural of Structural_Xor is
    signal wire1, wire2 : std_logic;
begin
    wire1 <= a and not b;
    wire2 <= not a and b;
    y <= wire1 or wire2;
end architecture structural;

library ieee; use ieee.std_logic_1164.all;
entity tb_Structural_Xor is end entity tb_Structural_Xor;
architecture sim of tb_Structural_Xor is
    signal a, b, y : std_logic := '0';
begin
    uut: entity work.Structural_Xor port map(a=>a,b=>b,y=>y);
    process begin
        a<='0'; b<='0'; wait for 10 ns;
        a<='0'; b<='1'; wait for 10 ns;
        a<='1'; b<='0'; wait for 10 ns;
        a<='1'; b<='1'; wait for 10 ns;
        wait;
    end process;
end architecture sim;

How this Code Works (Simplified)

  • XOR built from AND + NOT + OR gates using internal wire signals.

10. Finite State Machines (FSM)

An FSM moves through predefined states based on clock edges and inputs.

Traffic Light Analogy

Green → Yellow → Red → Green. Each state has a fixed behaviour. On each clock tick, the machine transitions to the next state based on its current state and inputs.

Example: Toggle FSM 
library ieee; use ieee.std_logic_1164.all;
entity Toggle_FSM is
    port ( clk : in std_logic; state : inout std_logic );
end entity Toggle_FSM;
architecture behavioral of Toggle_FSM is
    signal next_state : std_logic;
begin
    next_state <= not state;
    state <= (next_state and clk) or (state and not clk);
end architecture behavioral;

library ieee; use ieee.std_logic_1164.all;
entity tb_Toggle_FSM is end entity tb_Toggle_FSM;
architecture sim of tb_Toggle_FSM is
    signal clk, state : std_logic := '0';
begin
    uut: entity work.Toggle_FSM port map(clk=>clk,state=>state);
    process begin
        clk<='0'; wait for 10 ns; clk<='1'; wait for 10 ns;
        clk<='0'; wait for 10 ns; clk<='1'; wait for 10 ns;
        clk<='0'; wait for 10 ns; clk<='1'; wait for 10 ns;
        wait;
    end process;
end architecture sim;

How this Code Works (Simplified)

  • State toggles on every rising clock edge — simple 2-state FSM.

11. Packages & Libraries

Import reusable toolboxes of predefined types and functions.

Cookbook Analogy

Instead of deriving binary addition from scratch, import the standard cookbook (library ieee) and use its recipes (use ieee.std_logic_1164.all).

Example: Full Adder 
library ieee; use ieee.std_logic_1164.all;
entity Full_Adder is
    port ( a, b, cin : in std_logic; sum, cout : out std_logic );
end entity Full_Adder;
architecture behavioral of Full_Adder is
begin
    sum  <= (a xor b) xor cin;
    cout <= (a and b) or (cin and (a xor b));
end architecture behavioral;

library ieee; use ieee.std_logic_1164.all;
entity tb_Full_Adder is end entity tb_Full_Adder;
architecture sim of tb_Full_Adder is
    signal a, b, cin, sum, cout : std_logic := '0';
begin
    uut: entity work.Full_Adder port map(a=>a,b=>b,cin=>cin,sum=>sum,cout=>cout);
    process begin
        a<='0'; b<='0'; cin<='0'; wait for 10 ns;
        a<='1'; b<='1'; cin<='0'; wait for 10 ns;
        a<='1'; b<='1'; cin<='1'; wait for 10 ns;
        wait;
    end process;
end architecture sim;

How this Code Works (Simplified)

  • sum — XOR of all three inputs gives the 1-bit sum digit.
  • cout — Carry bit passed to the next adder stage.

12. Parameterization (Generics)

Generics let you parameterize a module so one design works for 8-bit, 16-bit, or 32-bit without rewriting the code.

T-Shirt Factory Analogy

One factory design with a SIZE parameter — pass in S/M/L/XL and it produces the right shirt. Pass WIDTH=8 or WIDTH=16 and your register module scales automatically.

Example: Parameterized Buffer 
library ieee; use ieee.std_logic_1164.all;
entity Generic_Buffer is
    generic ( G_VAL : std_logic := '1' );
    port ( a : in std_logic; y : out std_logic );
end entity Generic_Buffer;
architecture behavioral of Generic_Buffer is
begin
    y <= a and G_VAL;
end architecture behavioral;

library ieee; use ieee.std_logic_1164.all;
entity tb_Generic is end entity tb_Generic;
architecture sim of tb_Generic is
    signal a, y : std_logic := '0';
begin
    uut: entity work.Generic_Buffer
        generic map(G_VAL=>'1')
        port map(a=>a, y=>y);
    process begin
        a<='0'; wait for 10 ns;
        a<='1'; wait for 10 ns;
        wait;
    end process;
end architecture sim;

How this Code Works (Simplified)

  • generic map — Passes configuration values at instantiation time without changing the design entity.

13. Functions & Procedures

Subprograms group reusable logic to keep designs clean.

Kitchen Blender Analogy

Put in ingredients (inputs), press one button, get a smoothie (return value). A Function returns one value. A Procedure can modify multiple output signals.

Example: Custom XOR Function 
library ieee; use ieee.std_logic_1164.all;
entity Function_Demo is
    port ( a, b : in std_logic; y : out std_logic );
end entity Function_Demo;
architecture behavioral of Function_Demo is
    function my_xor(x, z : std_logic) return std_logic is
    begin
        return x xor z;
    end function my_xor;
begin
    y <= my_xor(a, b);
end architecture behavioral;

library ieee; use ieee.std_logic_1164.all;
entity tb_Function is end entity tb_Function;
architecture sim of tb_Function is
    signal a, b, y : std_logic := '0';
begin
    uut: entity work.Function_Demo port map(a=>a,b=>b,y=>y);
    process begin
        a<='0'; b<='0'; wait for 10 ns;
        a<='1'; b<='0'; wait for 10 ns;
        a<='1'; b<='1'; wait for 10 ns;
        wait;
    end process;
end architecture sim;

How this Code Works (Simplified)

  • function my_xor — Declares a reusable XOR logic block called every time we write my_xor(a, b).

14. Self-Checking Testbenches & Asserts

Assert statements automatically verify circuit outputs and raise errors if results are wrong.

Smoke Detector Analogy

You don't sit watching the oven all day — you install a smoke detector that alarms automatically. Assert checks a condition and raises an alarm (error message) when it fails.

Example: Assert-Based Verification 
library ieee; use ieee.std_logic_1164.all;
entity Gate_Checked is
    port ( a : in std_logic; y : out std_logic );
end entity Gate_Checked;
architecture behavioral of Gate_Checked is
begin
    y <= not a;
end architecture behavioral;

library ieee; use ieee.std_logic_1164.all;
entity tb_Assert_Demo is end entity tb_Assert_Demo;
architecture sim of tb_Assert_Demo is
    signal a, y : std_logic := '0';
begin
    uut: entity work.Gate_Checked port map(a=>a,y=>y);
    process begin
        a<='0'; wait for 10 ns;
        assert (y='1') report "FAIL: y should be 1 when a=0" severity error;
        a<='1'; wait for 10 ns;
        assert (y='0') report "FAIL: y should be 0 when a=1" severity error;
        wait;
    end process;
end architecture sim;

How this Code Works (Simplified)

  • assert (y='1') — Checks condition. Prints error message and fails if condition is false.

15. Custom Arrays & Records

Group related signals under a single composite type name.

Excel vs Business Card Analogy

  • Array — Excel column: all cells same type (e.g., 16 sensor bits).
  • Record — Business card: different fields grouped (name, phone, email).
Example: Signal Bus Routing 
library ieee; use ieee.std_logic_1164.all;
entity Data_Bus is
    port ( in_val : in std_logic; out_val : out std_logic );
end entity Data_Bus;
architecture behavioral of Data_Bus is
    signal bus_line : std_logic;
begin
    bus_line <= in_val;
    out_val  <= bus_line;
end architecture behavioral;

library ieee; use ieee.std_logic_1164.all;
entity tb_Data_Bus is end entity tb_Data_Bus;
architecture sim of tb_Data_Bus is
    signal in_val, out_val : std_logic := '0';
begin
    uut: entity work.Data_Bus port map(in_val=>in_val,out_val=>out_val);
    process begin
        in_val<='0'; wait for 10 ns;
        in_val<='1'; wait for 10 ns;
        wait;
    end process;
end architecture sim;

How this Code Works (Simplified)

  • Data flows through an internal bus_line signal representing a memory/data bus path.

16. Synthesis vs Simulation

Simulation code runs in software. Synthesis code gets burned into real hardware.

Video Game vs Real Building Analogy

  • Simulation = Video game — you can fly, teleport, use wait for 10 ns, read files.
  • Synthesis = Real building — must follow physics. No wait for delays. Use rising_edge(clk) for timing.

Golden Rules for Synthesis

  • Use rising_edge(clk) — never wait for X ns
  • Add reset lines — transistors power up randomly
  • Close all if/case paths — avoid accidental latches
  • Keep testbench code separate from design entities

17. Delay Timing Models

VHDL models gate propagation delays using inertial and transport delay keywords.

Furnace vs Water Hose Analogy

  • Inertial (default) — Furnace: short pulses don't heat it (glitches < delay are filtered out).
  • Transport — Water hose: every ripple passes through, just delayed in time.
Example: Delay Models 
library ieee; use ieee.std_logic_1164.all;
entity Delay_Demo is
    port ( x : in std_logic; y_in : out std_logic; y_tr : out std_logic );
end entity Delay_Demo;
architecture behavioral of Delay_Demo is
begin
    y_in <= x after 5 ns;
    y_tr <= transport x after 5 ns;
end architecture behavioral;

library ieee; use ieee.std_logic_1164.all;
entity tb_Delay_Demo is end entity tb_Delay_Demo;
architecture sim of tb_Delay_Demo is
    signal x, y_in, y_tr : std_logic := '0';
begin
    uut: entity work.Delay_Demo port map(x=>x,y_in=>y_in,y_tr=>y_tr);
    process begin
        x<='0'; wait for 10 ns;
        x<='1'; wait for 10 ns;
        x<='0'; wait for 2 ns;
        x<='1'; wait for 10 ns;
        wait;
    end process;
end architecture sim;

How this Code Works (Simplified)

  • after 5 ns — Inertial: 2 ns spike is rejected (shorter than 5 ns delay).
  • transport ... after 5 ns — All pulses pass, simply time-shifted.

18. Signal Attributes

Built-in query functions that inspect signal properties at runtime.

Key Attributes

  • clk'event and clk='1' — Rising edge detection
  • s'last_value — Previous value before last change
  • arr'length — Number of elements in an array
  • std_logic'image(q) — Convert value to printable string
Example: Attribute Usage 
library ieee; use ieee.std_logic_1164.all;
entity Attr_Monitor is
    port ( clk : in std_logic; q : out std_logic );
end entity Attr_Monitor;
architecture behavioral of Attr_Monitor is
begin
    q <= clk;
end architecture behavioral;

library ieee; use ieee.std_logic_1164.all;
entity tb_Attr_Monitor is end entity tb_Attr_Monitor;
architecture sim of tb_Attr_Monitor is
    signal clk, q : std_logic := '0';
begin
    uut: entity work.Attr_Monitor port map(clk=>clk,q=>q);
    process begin
        clk<='0'; wait for 10 ns;
        clk<='1'; wait for 10 ns;
        report "q = " & std_logic'image(q);
        wait;
    end process;

23. Coding Style Standards

Consistent coding style makes VHDL readable, maintainable, and synthesis-friendly.

24. Clock Domain Crossing & Metastability

When data crosses from one clock domain to another, it can cause metastability — the flip-flop output gets stuck between '0' and '1' for an unpredictable time.

Two Elevators Analogy

Two elevators on different schedules (different clock speeds). If you try to jump from one to the other mid-trip, you might land in the gap (metastability). The solution: wait at a synchroniser floor (a double-register synchroniser) between the two elevator systems.

Solution: Double-Flop Synchroniser

Example: 2-Stage Synchroniser 
library ieee; use ieee.std_logic_1164.all;
entity Synchroniser is
    port (
        clk_dst  : in  std_logic;
        data_in  : in  std_logic;
        data_out : out std_logic
    );
end entity Synchroniser;

architecture behavioral of Synchroniser is
    signal ff1, ff2 : std_logic := '0';
begin
    -- Two-flip-flop synchroniser to reduce metastability risk
    sync_proc: process(clk_dst)
    begin
        if rising_edge(clk_dst) then
            ff1      <= data_in;   -- Stage 1: capture
            ff2      <= ff1;       -- Stage 2: re-register
            data_out <= ff2;       -- Output stable version
        end if;
    end process sync_proc;
end architecture behavioral;

library ieee; use ieee.std_logic_1164.all;
entity tb_Synchroniser is end entity tb_Synchroniser;
architecture sim of tb_Synchroniser is
    signal clk     : std_logic := '0';
    signal data_in : std_logic := '0';
    signal data_out: std_logic;
begin
    uut: entity work.Synchroniser port map(clk_dst=>clk,data_in=>data_in,data_out=>data_out);
    -- Clock generator 10ns period
    clk <= not clk after 5 ns;
    process begin
        data_in <= '0'; wait for 20 ns;
        data_in <= '1'; wait for 20 ns;
        data_in <= '0'; wait for 20 ns;
        wait;
    end process;
end architecture sim;

How this Code Works (Simplified)

  • ff1 <= data_in — First flip-flop captures the async input (may be metastable).
  • ff2 <= ff1 — Second flip-flop gives ff1 time to settle, producing a stable output.
  • rising_edge(clk) — Synthesizable clock edge detection.

library ieee; use ieee.std_logic_1164.all;
entity DFF_Reset is
    port (
        clk   : in  std_logic;

end entity DFF_Reset;
architecture behavioral of DFF_Reset is
begin
    process(clk)
    begin
        if rising_edge(clk) then
            if rst='1' then
                q <= '0';     -- Synchronous reset
            else
                q <= d;
            end if;
        end if;
    end process;

use ieee.numeric_std.all;

32. Pipeline Design

Pipelining breaks a complex computation into stages — each stage is a registered layer, enabling high-speed operation.

Car Assembly Line Analogy

Instead of one worker building an entire car (slow), you have a conveyor belt where Worker 1 fits the engine, Worker 2 fits the doors, Worker 3 fits the wheels — all in parallel. Every clock tick, one car advances to the next stage and a new car enters Stage 1. Throughput = 1 car per tick despite each car taking 3 ticks total.

Example: 3-Stage Pipeline 
library ieee;
use ieee.std_logic_1164.all;
use ieee.numeric_std.all;

entity Pipeline3 is
    port (
        clk  : in  std_logic;
        rst  : in  std_logic;
        din  : in  std_logic_vector(3 downto 0);
        dout : out std_logic_vector(3 downto 0)
    );
end entity Pipeline3;

architecture behavioral of Pipeline3 is
    signal stage1, stage2 : std_logic_vector(3 downto 0) := (others=>'0');
begin
    process(clk)
    begin
        if rising_edge(clk) then
            if rst='1' then
                stage1 <= (others=>'0');
                stage2 <= (others=>'0');
                dout   <= (others=>'0');
            else
                stage1 <= din;              -- Stage 1 register
                stage2 <= stage1;           -- Stage 2 register
                dout   <= stage2;           -- Stage 3 output
            end if;
        end if;
    end process;
end architecture behavioral;

library ieee; use ieee.std_logic_1164.all;
entity tb_Pipeline3 is end entity tb_Pipeline3;
architecture sim of tb_Pipeline3 is
    signal clk  : std_logic := '0';
    signal rst  : std_logic := '1';
    signal din  : std_logic_vector(3 downto 0) := "0000";
    signal dout : std_logic_vector(3 downto 0);
begin
    uut: entity work.Pipeline3 port map(clk=>clk,rst=>rst,din=>din,dout=>dout);
    clk <= not clk after 5 ns;
    process begin
        rst <= '1'; wait for 20 ns;
        rst <= '0';
        din <= "0001"; wait for 10 ns;
        din <= "0010"; wait for 10 ns;
        din <= "0011"; wait for 10 ns;
        din <= "0100"; wait for 40 ns;

    end process;
end architecture sim;

How this Code Works (Simplified)

  • Data enters Stage 1, advances to Stage 2, then exits at Stage 3 — each stage adds exactly 1 clock cycle of latency.
  • Total latency = 3 clock cycles. Throughput = 1 new result per clock cycle.

33. FIFO Queue Design

A FIFO (First-In, First-Out) buffer temporarily stores data between two domains running at different speeds.

Queue at Bank Analogy

Customers join the queue (write port) and are served in arrival order (read port). The queue fills if customers arrive faster than they're served (overflow), and empties if nobody arrives (underflow). FIFO status flags (full/empty) prevent both conditions.

Example: 4-Deep FIFO 
library ieee;
use ieee.std_logic_1164.all;
use ieee.numeric_std.all;

entity FIFO4 is
    port (
        clk   : in  std_logic;
        rst   : in  std_logic;
        wr_en : in  std_logic;
        rd_en : in  std_logic;
        din   : in  std_logic_vector(3 downto 0);
        dout  : out std_logic_vector(3 downto 0);
        full  : out std_logic;
        empty : out std_logic
    );
end entity FIFO4;

architecture behavioral of FIFO4 is
    type fifo_mem is array(0 to 3) of std_logic_vector(3 downto 0);
    signal mem   : fifo_mem := (others => (others => '0'));
    signal wr_ptr, rd_ptr : unsigned(1 downto 0) := (others => '0');
    signal count : unsigned(2 downto 0) := (others => '0');
begin
    full  <= '1' when count = 4 else '0';
    empty <= '1' when count = 0 else '0';
    dout  <= mem(to_integer(rd_ptr));

    process(clk)
    begin
        if rising_edge(clk) then
            if rst='1' then
                wr_ptr <= (others=>'0');
                rd_ptr <= (others=>'0');
                count  <= (others=>'0');
            else
                if wr_en='1' and count < 4 then
                    mem(to_integer(wr_ptr)) <= din;
                    wr_ptr <= wr_ptr + 1;
                    count  <= count + 1;
                end if;
                if rd_en='1' and count > 0 then
                    rd_ptr <= rd_ptr + 1;
                    count  <= count - 1;
                end if;
            end if;
        end if;
    end process;
end architecture behavioral;

library ieee; use ieee.std_logic_1164.all;
entity tb_FIFO4 is end entity tb_FIFO4;
architecture sim of tb_FIFO4 is
    signal clk, rst, wr_en, rd_en, full, empty : std_logic := '0';
    signal din, dout : std_logic_vector(3 downto 0) := "0000";
begin
    uut: entity work.FIFO4 port map(clk=>clk,rst=>rst,wr_en=>wr_en,rd_en=>rd_en,din=>din,dout=>dout,full=>full,empty=>empty);
    clk <= not clk after 5 ns;
    process begin
        rst<='1'; wait for 20 ns; rst<='0';
        wr_en<='1'; din<="0001"; wait for 10 ns;
        din<="0010"; wait for 10 ns;
        wr_en<='0'; rd_en<='1'; wait for 20 ns;
        rd_en<='0'; wait for 10 ns;
        wait;
    end process;
end architecture sim;

How this Code Works (Simplified)

  • wr_ptr / rd_ptr — Write and read pointers advance independently like two runners on a circular track.
  • count — Tracks how many items are currently in the FIFO.
  • full / empty — Status flags prevent overflow and underflow.

34. SPI / I2C Protocol Modeling

SPI and I2C are serial communication protocols used to connect FPGA with sensors, displays, ADCs, and other chips.

Walkie-Talkie Analogy

  • SPI — Walkie-talkie with a push-to-talk button (SCLK) and separate transmit/receive channels (MOSI/MISO). Fast, full-duplex, 4-wire.
  • I2C — Party phone line: one clock (SCL) and one shared data wire (SDA) where multiple devices share the same line using unique 7-bit addresses.
Example: SPI Master Transmitter 
library ieee;
use ieee.std_logic_1164.all;
use ieee.numeric_std.all;

entity SPI_Master is
    port (
        clk   : in  std_logic;
        rst   : in  std_logic;
        start : in  std_logic;
        data  : in  std_logic_vector(7 downto 0);
        sclk  : out std_logic;
        mosi  : out std_logic;
        cs_n  : out std_logic;
        done  : out std_logic
    );
end entity SPI_Master;

architecture behavioral of SPI_Master is
    signal shift_reg : std_logic_vector(7 downto 0) := (others=>'0');
    signal bit_cnt   : unsigned(3 downto 0) := (others=>'0');
    signal tx_active : std_logic := '0';
    signal clk_div   : std_logic := '0';
begin
    sclk <= clk_div;
    mosi <= shift_reg(7);
    cs_n <= not tx_active;

    process(clk)
    begin
        if rising_edge(clk) then
            if rst='1' then
                tx_active <= '0'; bit_cnt <= (others=>'0');
                clk_div   <= '0'; done    <= '0';
            elsif start='1' and tx_active='0' then
                shift_reg <= data; tx_active <= '1';
                bit_cnt   <= (others=>'0'); done <= '0';
            elsif tx_active='1' then
                clk_div <= not clk_div;
                if clk_div='1' then
                    shift_reg <= shift_reg(6 downto 0) & '0';
                    bit_cnt   <= bit_cnt + 1;
                    if bit_cnt = 7 then
                        tx_active <= '0'; done <= '1';
                    end if;
                end if;
            else
                done <= '0';
            end if;
        end if;
    end process;
end architecture behavioral;

library ieee; use ieee.std_logic_1164.all;
entity tb_SPI is end entity tb_SPI;
architecture sim of tb_SPI is
    signal clk, rst, start, sclk, mosi, cs_n, done : std_logic := '0';
    signal data : std_logic_vector(7 downto 0) := "10110100";
begin
    uut: entity work.SPI_Master port map(clk=>clk,rst=>rst,start=>start,data=>data,sclk=>sclk,mosi=>mosi,cs_n=>cs_n,done=>done);

How this Code Works (Simplified)

  • shift_reg(7) — MSB of shift register is sent on MOSI, one bit per clock.
  • cs_n — Chip select goes low when transmission starts (activates the slave device).
  • bit_cnt = 7 — After 8 bits, transmission complete and done flag raised.

35. UART Transmitter & Receiver

UART (Universal Asynchronous Receiver/Transmitter) sends data serially over a single wire at a fixed baud rate — used in debug consoles, GPS, Bluetooth modules.

Telegram Analogy

Telegram messages are sent letter by letter (bit by bit) over a wire. A start code (start bit) signals the beginning. Each letter follows at a fixed speed (baud rate). A stop code ends the message. No shared clock — sender and receiver agree on speed in advance.

Example: UART Transmitter 
library ieee;
use ieee.std_logic_1164.all;
use ieee.numeric_std.all;

entity UART_TX is
    generic ( CLKS_PER_BIT : integer := 10 );
    port (
        clk      : in  std_logic;
        rst      : in  std_logic;
        tx_start : in  std_logic;
        tx_data  : in  std_logic_vector(7 downto 0);
        tx_pin   : out std_logic;
        tx_done  : out std_logic
    );
end entity UART_TX;

architecture behavioral of UART_TX is
    type t_state is (IDLE, START, DATA, STOP);
    signal state    : t_state := IDLE;
    signal clk_cnt  : integer range 0 to CLKS_PER_BIT-1 := 0;
    signal bit_idx  : integer range 0 to 7 := 0;
    signal shift_r  : std_logic_vector(7 downto 0) := (others=>'0');
begin
    process(clk)
    begin
        if rising_edge(clk) then
            if rst='1' then
                state <= IDLE; tx_pin <= '1'; tx_done <= '0';
            else
                tx_done <= '0';
                case state is
                    when IDLE =>
                        tx_pin <= '1';
                        if tx_start='1' then
                            shift_r <= tx_data; clk_cnt <= 0; state <= START;
                        end if;
                    when START =>
                        tx_pin <= '0';
                        if clk_cnt = CLKS_PER_BIT-1 then
                            clk_cnt <= 0; bit_idx <= 0; state <= DATA;
                        else clk_cnt <= clk_cnt + 1; end if;
                    when DATA =>
                        tx_pin <= shift_r(bit_idx);
                        if clk_cnt = CLKS_PER_BIT-1 then
                            clk_cnt <= 0;
                            if bit_idx = 7 then state <= STOP;
                            else bit_idx <= bit_idx + 1; end if;
                        else clk_cnt <= clk_cnt + 1; end if;
                    when STOP =>
                        tx_pin <= '1';
                        if clk_cnt = CLKS_PER_BIT-1 then
                            tx_done <= '1'; state <= IDLE;
                        else clk_cnt <= clk_cnt + 1; end if;
                end case;
            end if;
        end if;
    end process;
end architecture behavioral;

library ieee; use ieee.std_logic_1164.all;
entity tb_UART_TX is end entity tb_UART_TX;
architecture sim of tb_UART_TX is
    signal clk, rst, tx_start, tx_pin, tx_done : std_logic := '0';
    signal tx_data : std_logic_vector(7 downto 0) := "01000001";
begin
    uut: entity work.UART_TX generic map(CLKS_PER_BIT=>10)
        port map(clk=>clk,rst=>rst,tx_start=>tx_start,tx_data=>tx_data,tx_pin=>tx_pin,tx_done=>tx_done);
    clk <= not clk after 5 ns;

    end process;
end architecture sim;

How this Code Works (Simplified)

  • FSM states: IDLE → START bit → 8 DATA bits → STOP bit → back to IDLE.
  • CLKS_PER_BIT — Generic parameter sets baud rate (actual = clk_freq / baud_rate).
  • tx_pin — Idles HIGH, goes LOW for start bit, then sends bits LSB-first.

36. PWM Signal Generator

PWM (Pulse Width Modulation) controls the average power delivered to a load by rapidly switching a signal on/off at a controlled duty cycle.

Dimmer Switch Analogy

A ceiling fan dimmer switch doesn't reduce voltage — it rapidly switches power on/off hundreds of times per second. When on 70% of the time (70% duty cycle), the fan spins at 70% speed. PWM works the same way for LED brightness, motor speed, and servo control.

Example: PWM Generator 
library ieee;
use ieee.std_logic_1164.all;
use ieee.numeric_std.all;

entity PWM_Gen is
    generic ( PERIOD : integer := 20 );  -- PWM period in clock cycles
    port (
        clk      : in  std_logic;
        rst      : in  std_logic;
        duty     : in  integer range 0 to 20;  -- Duty cycle count
        pwm_out  : out std_logic
    );
end entity PWM_Gen;

architecture behavioral of PWM_Gen is
    signal counter : integer range 0 to 20 := 0;
begin
    process(clk)
    begin
        if rising_edge(clk) then
            if rst='1' then
                counter <= 0; pwm_out <= '0';
            else
                if counter < PERIOD-1 then
                    counter <= counter + 1;
                else
                    counter <= 0;
                end if;
                if counter < duty then
                    pwm_out <= '1';   -- High portion
                else
                    pwm_out <= '0';   -- Low portion
                end if;
            end if;
        end if;
    end process;
end architecture behavioral;

library ieee; use ieee.std_logic_1164.all;
entity tb_PWM is end entity tb_PWM;
architecture sim of tb_PWM is
    signal clk, rst, pwm_out : std_logic := '0';
    signal duty : integer := 10;  -- 50% duty cycle
begin
    uut: entity work.PWM_Gen generic map(PERIOD=>20)
        port map(clk=>clk,rst=>rst,duty=>duty,pwm_out=>pwm_out);
    clk <= not clk after 5 ns;
    process begin
        rst<='1'; wait for 20 ns; rst<='0';
        duty <= 10; wait for 200 ns;   -- 50% duty
        duty <= 15; wait for 200 ns;   -- 75% duty
        duty <= 5;  wait for 200 ns;   -- 25% duty
        wait;
    end process;
end architecture sim;

How this Code Works (Simplified)

  • counter < duty — Output is HIGH during duty cycle count, LOW after it.
  • duty=10, PERIOD=20 — 50% duty cycle (10 clocks high, 10 clocks low).
  • Change duty at runtime to adjust brightness/speed without modifying the hardware design.

37. Coverage-Driven Verification

Coverage measures how thoroughly your testbench has exercised all possible circuit states and paths.

Checklist Analogy

A flight pre-departure checklist has 50 items. Coverage = percentage of items checked. You don't declare the plane ready to fly at 40% coverage! Similarly, a chip is not declared verified until all code paths, states, and transitions are covered.

Types of Coverage

  • Code Coverage — Has every VHDL line been executed at least once?
  • Branch Coverage — Has every if/else path and case branch been taken?
  • Toggle Coverage — Has every signal toggled between '0' and '1' at least once?
  • Functional Coverage — Have all meaningful input combinations and state transitions occurred?
Example: Full Truth Table Coverage 
library ieee; use ieee.std_logic_1164.all;
entity Gate_Coverage is
    port ( a, b : in std_logic; y_and, y_or, y_xor : out std_logic );
end entity Gate_Coverage;
architecture behavioral of Gate_Coverage is
begin
    y_and <= a and b;
    y_or  <= a or  b;
    y_xor <= a xor b;
end architecture behavioral;

library ieee; use ieee.std_logic_1164.all;

architecture sim of tb_Coverage is

begin

        -- Full truth table - 100% input combination coverage
        a<='0'; b<='0'; wait for 10 ns;
        assert y_and='0' and y_or='0' and y_xor='0' report "FAIL 00" severity error;
        a<='0'; b<='1'; wait for 10 ns;
        assert y_and='0' and y_or='1' and y_xor='1' report "FAIL 01" severity error;
        a<='1'; b<='0'; wait for 10 ns;

        a<='1'; b<='1'; wait for 10 ns;

  • All 4 input combinations (00,01,10,11) are tested with assert verification — achieving 100% input coverage for a 2-input gate.

Instead of manually writing every test case, generate random inputs with constraints to automatically discover edge cases.

Dice with Rules Analogy

Rolling dice gives random numbers, but rules constrain them (e.g., "only even numbers", "sum must be < 10"). Constrained random testing = random inputs within meaningful boundaries, automatically finding bugs manual tests miss.

39. UVM Concepts in VHDL

UVM (Universal Verification Methodology) is the industry-standard framework for structured verification, primarily used with SystemVerilog but its concepts apply in VHDL too.

  • Sequencer = Conveyor belt feeding test items to testers
  • Driver = Robot arm applying specific forces/inputs to the DUT

41. FPGA Resource Utilization

FPGAs have fixed hardware resources. Understanding and optimizing resource usage is essential for large designs.

Office Space Analogy

An FPGA is an office building with a fixed number of desks (LUTs), filing cabinets (flip-flops), meeting rooms (BRAMs), and calculators (DSP blocks). Your VHDL code determines how many of each resource your design occupies. Fitting more employees (logic) than desks causes compilation failure.

Key FPGA Resources

  • LUT (Look-Up Table) — Implements combinational logic. A 6-input LUT can implement any function of 6 variables.
  • Flip-Flop (FF) — Stores one bit of state. One per LUT slice on most architectures.
  • BRAM (Block RAM) — Dedicated memory blocks (18Kb or 36Kb). Far more efficient than building RAM from LUTs.
  • DSP Block — Hardware multiplier-accumulator (18x18 multiply). Use for signal processing; much faster than LUT-based multiplication.
  • I/O Pins — Limited number of physical pins. Route high-speed signals to high-performance I/O banks.

Optimization Tips

  • Use unsigned/signed arithmetic to infer DSP blocks instead of custom LUT multipliers.
  • Use type ram_t is array patterns to infer BRAMs instead of register arrays.
  • Share resources between mutually exclusive operations.
  • Pipeline critical paths to reduce logic depth and improve Fmax.

42. Timing Constraints (SDC)

Timing constraints tell the synthesis tool how fast your circuit must run, enabling it to place and route logic to meet speed requirements.

Speed Limit Analogy

Road speed limits (timing constraints) tell drivers how fast they're allowed to go. The synthesis tool is the driver — it optimises route placement to ensure signals arrive within the speed limit (setup time requirement). Without constraints, the tool doesn't know what speed is needed and may produce a slow, unsafe design.

Key Timing Concepts

  • Setup Time — Data must be stable at D-input BEFORE the clock edge arrives.
  • Hold Time — Data must remain stable AFTER the clock edge.
  • Slack — Timing margin. Positive slack = passes timing. Negative slack = timing violation (chip may malfunction).
  • Fmax — Maximum operating frequency the design can achieve after place-and-route.

Basic SDC Constraint Example

# Create 100 MHz system clock (10ns period) on pin clk
create_clock -period 10.0 -name sys_clk [get_ports clk]

# Set input arrival time for data ports (relative to clock)
set_input_delay -clock sys_clk 2.0 [get_ports data_in]

# Set output required time for output ports
set_output_delay -clock sys_clk 2.0 [get_ports data_out]
Example: Timing-Aware Register Design

44. Power Optimization Techniques

FPGA and ASIC designs consume power through dynamic switching and static leakage. Power optimization is critical for battery-powered IoT devices and high-density chips.

Energy Saving Mode Analogy

A smartphone has power-saving mode — it dims the screen (clock gating), pauses background apps (operand isolation), and throttles the CPU (dynamic voltage/frequency scaling). VHDL power optimization uses the same ideas at the hardware level.

Key Power Optimization Techniques

  • Clock Gating — Disable the clock to unused registers when they don't need to update. Saves 20-40% dynamic power. 
    -- Inferred clock gate: tool inserts an AND gate on the clock
process(clk)
begin
    if rising_edge(clk) then
        if enable='1' then   -- Only updates when enabled
            reg <= data_in;
        end if;
    end if;
end process;
  • Operand Isolation — Prevent unused datapath inputs from toggling (each toggle costs switching energy). 
    -- Gate the input to a multiplier when not needed:
mult_in <= data when enable='1' else (others=>'0');
  • Reduce Toggle Rate — High-frequency signals consume more power. Use the lowest clock frequency that meets timing requirements.
  • Multi-Voltage Design — Run low-performance blocks at lower supply voltage (lower dynamic power ∝ V²).
  • Power Domains — Shut down entire design blocks when idle (used in mobile SoCs).
Example: Clock-Gated Register 
library ieee; use ieee.std_logic_1164.all;
entity Gated_Reg is
    port (
        clk    : in  std_logic;
        rst    : in  std_logic;
        enable : in  std_logic;   -- Clock gate enable
        d      : in  std_logic;
        q      : out std_logic
    );
end entity Gated_Reg;

architecture rtl of Gated_Reg is
begin
    process(clk)
    begin
        if rising_edge(clk) then
            if rst='1' then
                q <= '0';
            elsif enable='1' then   -- Only updates when enabled (saves power)
                q <= d;
            end if;
            -- When enable='0': register retains value, no switching activity
        end if;
    end process;
end architecture rtl;

library ieee; use ieee.std_logic_1164.all;
entity tb_Gated_Reg is end entity tb_Gated_Reg;
architecture sim of tb_Gated_Reg is
    signal clk, rst, enable, d, q : std_logic := '0';
begin
    uut: entity work.Gated_Reg port map(clk=>clk,rst=>rst,enable=>enable,d=>d,q=>q);
    clk <= not clk after 5 ns;
    process begin
        rst<='1'; wait for 20 ns; rst<='0';
        enable<='0'; d<='1'; wait for 30 ns;  -- q stays '0' (power saved)
        enable<='1'; wait for 20 ns;            -- q captures d='1'
        enable<='0'; d<='0'; wait for 30 ns;  -- q stays '1' (power saved)
        wait;
    end process;
end architecture sim;

How this Code Works (Simplified)

  • When enable='0': the register doesn't update — no signal toggling = zero dynamic power consumption in this register.
  • Synthesis tools recognize this pattern and insert an efficient hardware clock gate instead of routing the enable through the data path.

45. Enumerated Types & FSM Encoding

VHDL enumerated types allow developers to use custom state names. Hardware compilers map these to binary representation logic using encoding schemes.

Secret Codes Analogy

A spy network uses secret handshakes. One-Hot encoding uses one flag per state (1000, 0100, 0010, 0001). Gray Code changes only a single bit between consecutive numbers (00, 01, 11, 10), preventing transitions spikes. Binary encoding uses the standard index value (00, 01, 10, 11).

Example: Address Data Copy DMA 
library ieee;
use ieee.std_logic_1164.all;

entity Dma_Controller is
    port (
        clk       : in  std_logic;
        rst       : in  std_logic;
        start     : in  std_logic;
        mem_data  : in  std_logic_vector(7 downto 0);
        mem_write : out std_logic;
        dma_busy  : out std_logic
    );
end entity Dma_Controller;

architecture behavioral of Dma_Controller is
    type state_t is (IDLE, READ_SRC, WRITE_DST, DONE);
    signal current_state : state_t := IDLE;
    signal temp_buffer   : std_logic_vector(7 downto 0) := (others => '0');
begin
    process(clk)
    begin
        if rising_edge(clk) then
            if rst = '1' then
                current_state <= IDLE;
                mem_write     <= '0';
                dma_busy      <= '0';
            else
                case current_state is
                    when IDLE =>
                        if start = '1' then
                            dma_busy      <= '1';
                            current_state <= READ_SRC;
                        end if;
                    when READ_SRC =>
                        -- Capture source address contents
                        temp_buffer   <= mem_data;
                        current_state <= WRITE_DST;
                    when WRITE_DST =>
                        -- Trigger write enable pulse to destination address
                        mem_write     <= '1';
                        current_state <= DONE;
                    when DONE =>
                        mem_write     <= '0';
                        dma_busy      <= '0';
                        current_state <= IDLE;
                end case;
            end if;
        end if;
    end process;
end architecture behavioral;

72. Simple 8-Bit CPU Design

Building a CPU controller featuring a Program Counter, Instruction Register, Accumulator, and Arithmetic Logic Unit.

8-Bit Processor Logic

Example: Accumulator CPU 
library ieee;
use ieee.std_logic_1164.all;
use ieee.numeric_std.all;

entity Simple_Cpu is
    port (
        clk  : in  std_logic;
        rst  : in  std_logic;
        code : in  std_logic_vector(7 downto 0); -- OP-Code Instruction bus
        accum: out std_logic_vector(7 downto 0)  -- Accumulator register output
    );
end entity Simple_Cpu;

architecture rtl of Simple_Cpu is
    signal acc_reg : unsigned(7 downto 0) := (others => '0');
    signal pc      : integer range 0 to 15 := 0;
begin
    accum <= std_logic_vector(acc_reg);

    process(clk)
        alias op_code : std_logic_vector(3 downto 0) is code(7 downto 4);
        alias operand : std_logic_vector(3 downto 0) is code(3 downto 0);
    begin
        if rising_edge(clk) then
            if rst = '1' then
                acc_reg <= (others => '0');
                pc      <= 0;
            else
                pc <= pc + 1; -- Increment Program Counter address
                -- Decode Opcode commands:
                -- "0001": ADD operand value to accumulator
                -- "0010": SUBTRACT operand value from accumulator
                if op_code = "0001" then
                    acc_reg <= acc_reg + unsigned(operand);
                elsif op_code = "0010" then
                    acc_reg <= acc_reg - unsigned(operand);
                end if;
            end if;
        end if;
    end process;
end architecture rtl;

library ieee; use ieee.std_logic_1164.all;
entity tb_Cpu is end entity tb_Cpu;
architecture sim of tb_Cpu is
    signal clk, rst : std_logic := '0';
    signal code  : std_logic_vector(7 downto 0) := "00000000";
    signal accum : std_logic_vector(7 downto 0);
begin
    uut: entity work.Simple_Cpu port map(clk=>clk, rst=>rst, code=>code, accum=>accum);
    clk <= not clk after 5 ns;
    process begin
        rst <= '1'; wait for 20 ns;
        rst <= '0';
        -- Add 5 to Acc -> "0001" (ADD) & "0101" (operand 5)
        code <= "00010101"; wait for 10 ns;
        -- Subtract 2 from Acc -> "0010" (SUB) & "0010" (operand 2)
        code <= "00100010"; wait for 10 ns;
        code <= "00000000"; wait for 20 ns;
        wait;
    end process;
end architecture sim;

How this Code Works (Simplified)

  • op_code / operand — Splits the 8-bit instruction into controller commands (bits 7–4) and numerical variables (bits 3–0).
  • acc_reg <= acc_reg + unsigned(operand) — Performs addition math and stores the running total in register memory.
