1. VHDL Introduction

VHDL stands for VHSIC (Very High Speed Integrated Circuit) Hardware Description Language. It is a standard language defined by IEEE used to design, model, and simulate digital electronic systems ranging from basic logic gates to complex microprocessors.

Hardware vs Software: The Switch Analogy

If you have never coded before, think of normal software (like Python or Java) as a cooking recipe: the computer reads line 1, executes it, then moves to line 2, executing one instruction at a time in order.

VHDL is different. It describes physical hardware wiring. When you write VHDL, you are acting as an electrician laying down copper wires and switches. All wires carry electricity and operate at the exact same time (in parallel). Flipping a switch in a house instantly turns on a light bulb; it does not wait for other rooms to finish. VHDL works exactly like that!

Example: Half Adder & Testbench 
library ieee;
use ieee.std_logic_1164.all;

entity Half_Adder is
    port (
        a, b : in std_logic;
        sum, carry : out std_logic
    );
end entity Half_Adder;

architecture behavioral of Half_Adder is
begin
    sum <= a xor b;
    carry <= a and b;
end architecture behavioral;

library ieee;
use ieee.std_logic_1164.all;

entity tb_Half_Adder is
end entity tb_Half_Adder;

architecture sim of tb_Half_Adder is
    signal a, b : std_logic := '0';
    signal sum, carry : std_logic;
begin
    -- Instantiate the Unit Under Test (UUT)
    uut: entity work.Half_Adder
        port map (
            a => a,
            b => b,
            sum => sum,
            carry => carry
        );

    -- Stimulus process
    stim_proc: process
    begin
        -- Test Case 1: a=0, b=0
        a <= '0'; b <= '0';
        wait for 10 ns;
        report "Test 1: a=0, b=0 -> sum=" & std_logic'image(sum) & ", carry=" & std_logic'image(carry);

        -- Test Case 2: a=0, b=1
        a <= '0'; b <= '1';
        wait for 10 ns;
        report "Test 2: a=0, b=1 -> sum=" & std_logic'image(sum) & ", carry=" & std_logic'image(carry);

        -- Test Case 3: a=1, b=0
        a <= '1'; b <= '0';
        wait for 10 ns;
        report "Test 3: a=1, b=0 -> sum=" & std_logic'image(sum) & ", carry=" & std_logic'image(carry);

        -- Test Case 4: a=1, b=1
        a <= '1'; b <= '1';
        wait for 10 ns;
        report "Test 4: a=1, b=1 -> sum=" & std_logic'image(sum) & ", carry=" & std_logic'image(carry);

        wait;
    end process;
end architecture sim;

How this Code Works (Simplified)

  • library ieee; - Tells VHDL to import standard digital engineering packages (toolboxes) containing predefined data types.
  • entity Half_Adder - Defines the "box interface" of our chip container. It specifies two input wires (`a`, `b`) coming in, and two output wires (`sum`, `carry`) going out.
  • architecture behavioral - Defines the internal copper wiring of the chip using logical rules (`xor` and `and`).
  • tb_Half_Adder & process - This is our simulation workbench. It feeds inputs into the chip and delays time (`wait for 10 ns`) to test that the outputs compute correctly.

2. VHDL Entity & Architecture

A VHDL design is split into two primary components: an **Entity** and an **Architecture**.

The TV Remote Analogy

Think of a TV remote control:

  • The Entity: The plastic shell and buttons (Power, Vol+, Vol-, Ch). It defines the physical interface pins (buttons) you can press, but does not explain how it works inside.
  • The Architecture: The circuit board and microcontroller inside the remote that translates button presses into infrared signals. It defines the actual functionality.
Example: 2-to-1 Multiplexer Circuit 
library ieee;
use ieee.std_logic_1164.all;

entity Mux21 is
    port (
        sel, d0, d1 : in std_logic;
        y           : out std_logic
    );
end entity Mux21;

architecture behavioral of Mux21 is
begin
    -- Parallel Boolean logic mapping selector logic
    y <= (not sel and d0) or (sel and d1);
end architecture behavioral;

library ieee;
use ieee.std_logic_1164.all;

entity tb_Mux21 is
end entity tb_Mux21;

architecture sim of tb_Mux21 is
    signal sel : std_logic := '0';
    signal d0 : std_logic := '1';
    signal d1 : std_logic := '0';
    signal y : std_logic;
begin
    uut: entity work.Mux21
        port map (
            sel => sel,
            d0 => d0,
            d1 => d1,
            y => y
        );

    process
    begin
        sel <= '0'; wait for 10 ns;
        sel <= '1'; wait for 10 ns;
        d0 <= '0'; d1 <= '1'; sel <= '0'; wait for 10 ns;
        sel <= '1'; wait for 10 ns;
        wait;
    end process;
end architecture sim;

How this Code Works (Simplified)

  • entity Mux21 - Declares a selection pin chip with inputs (`sel`, `d0`, `d1`) and output (`y`).
  • y <= (not sel and d0) or (sel and d1) - Internal wiring. When selection line `sel` is low ('0'), output `y` is connected to input `d0`. When `sel` is high ('1'), `y` is connected to `d1`.

3. VHDL Signals vs Variables

In VHDL, data is routed and stored using either **Signals** or **Variables**.

The Conveyor Belt Analogy

  • Signals (<=): Like a **conveyor belt**. When you place a package on a conveyor belt, it takes time to travel from one end to another. Signals represent physical wires on a board and update only at the end of a simulation cycle.
  • Variables (:=): Like a **worker's notepad**. If the worker writes a number on a notepad, they can change or erase it instantly. Variables update immediately and are only used for temporary helper calculations inside process blocks.
Example: Intermediate Signals 
library ieee;
use ieee.std_logic_1164.all;

entity Logic_Circuit is
    port (
        a, b, c : in std_logic;
        out_y   : out std_logic
    );
end entity Logic_Circuit;

architecture behavioral of Logic_Circuit is
    -- Signal declarations act as local interconnect wires
    signal wire1 : std_logic;
    signal wire2 : std_logic;
begin
    wire1 <= a and b;
    wire2 <= not c;
    out_y <= wire1 or wire2;
end architecture behavioral;

library ieee;
use ieee.std_logic_1164.all;

entity tb_Logic_Circuit is
end entity tb_Logic_Circuit;

architecture sim of tb_Logic_Circuit is
    signal a, b, c : std_logic := '0';
    signal out_y : std_logic;
begin
    uut: entity work.Logic_Circuit
        port map (
            a => a,
            b => b,
            c => c,
            out_y => out_y
        );

    process
    begin
        a <= '0'; b <= '0'; c <= '0';
        wait for 10 ns;
        a <= '1'; b <= '1'; c <= '0';
        wait for 10 ns;
        a <= '0'; b <= '1'; c <= '1';
        wait for 10 ns;
        wait;
    end process;
end architecture sim;

How this Code Works (Simplified)

  • signal wire1, wire2 - Declares internal copper wiring paths inside our chip architecture.
  • wire1 <= a and b - Connects input lines `a` and `b` to an AND gate and puts the output on internal cable `wire1`.
  • out_y <= wire1 or wire2 - Connects internal lines `wire1` and `wire2` to an OR gate, sending the final voltage to output wire `out_y`.

4. VHDL Data Types

VHDL is a strongly typed language. This means you must declare the size and type of each wire or data path beforehand to prevent invalid connections.

Everyday Logic Gate Analogies

  • AND Gate: Cinema entry. You must have a ticket **AND** arrive on time (both conditions must be true).
  • OR Gate: Payment options. You pay with Cash **OR** pay with Card (either option is true).
  • NOT Gate: Toggle switch. If the input is off ('0'), the output is NOT off, i.e., on ('1').
  • XOR Gate: Soup or Salad choice. You can choose soup **OR** salad as a starter, but not both (inputs must be different).

Road vs Highway Analogy

  • std_logic (Single Bit) - Like a **one-lane road** where only one car (bit '0' or '1') can travel.
  • std_logic_vector (Bus) - Like a **multi-lane highway** where multiple bits travel in parallel, forming binary numbers (e.g. 8-bit, 16-bit).
Example: 3-Input OR Logic 
library ieee;
use ieee.std_logic_1164.all;

entity Or3_Gate is
    port (
        x, y, z : in std_logic;
        out_f   : out std_logic
    );
end entity Or3_Gate;

architecture behavioral of Or3_Gate is
begin
    out_f <= x or y or z;
end architecture behavioral;

library ieee;
use ieee.std_logic_1164.all;

entity tb_Or3 is
end entity tb_Or3;

architecture sim of tb_Or3 is
    signal x, y, z : std_logic := '0';
    signal out_f : std_logic;
begin
    uut: entity work.Or3_Gate
        port map (
            x => x,
            y => y,
            z => z,
            out_f => out_f
        );

    process
    begin
        x <= '0'; y <= '0'; z <= '0'; wait for 10 ns;
        x <= '1'; y <= '0'; z <= '0'; wait for 10 ns;
        x <= '0'; y <= '1'; z <= '1'; wait for 10 ns;
        wait;
    end process;
end architecture sim;

How this Code Works (Simplified)

  • x, y, z : in std_logic; - Declares three input pins, each holding a single bit value.
  • out_f <= x or y or z - Feeds the three pins into a 3-input OR gate. If any input is turned high ('1'), the output wire `out_f` turns on.

5. VHDL Concurrent Statements

In VHDL, any code written outside process blocks is **concurrent**. This is the native state of hardware circuitry.

The Water Pipes Analogy

Think of water pipes in your home. Turning on the kitchen sink tap instantly lets water flow. Turning on the bathroom shower tap does not wait for the kitchen sink to finish. Both pipes carry water **at the same time (in parallel)**.

Similarly, all concurrent assignments in VHDL execute in parallel. The physical electrical layout remains identical regardless of the order of statements in your code file.

Example: Comparator Circuit 
library ieee;
use ieee.std_logic_1164.all;

entity Comparator is
    port (
        a, b   : in std_logic;
        equals : out std_logic
    );
end entity Comparator;

architecture behavioral of Comparator is
begin
    -- Concurrent assignment (behaves as continuous connection)
    equals <= not (a xor b);
end architecture behavioral;

library ieee;
use ieee.std_logic_1164.all;

entity tb_Comparator is
end entity tb_Comparator;

architecture sim of tb_Comparator is
    signal a, b : std_logic := '0';
    signal equals : std_logic;
begin
    uut: entity work.Comparator
        port map (
            a => a,
            b => b,
            equals => equals
        );

    process
    begin
        a <= '0'; b <= '0';
        wait for 10 ns;
        a <= '0'; b <= '1';
        wait for 10 ns;
        a <= '1'; b <= '1';
        wait for 10 ns;
        wait;
    end process;
end architecture sim;

How this Code Works (Simplified)

  • equals <= not (a xor b) - Wiring of an XNOR gate. It compares inputs `a` and `b` continuously. If they are equal (both '0' or both '1'), output wire `equals` turns high ('1'). Otherwise, it goes low ('0').

6. VHDL Process Statements

A process block is a container for sequential statements. Inside a process, code executes line-by-line in order, mimicking standard software programming.

The Security Guard Analogy

Think of a process as a **security guard** sitting at a gate with a checklist of trigger variables (called the **sensitivity list**, e.g., `process(clk, reset)`):

  • The guard sleeps as long as the signals in the sensitivity list do not change.
  • The moment the clock pin `clk` or reset pin `reset` changes, the guard wakes up.
  • The guard executes all statements inside the process step-by-step from top to bottom.
  • Once finished, the guard sits back down and sleeps until the next change.
Example: Sequential Testbench Processes 
library ieee;
use ieee.std_logic_1164.all;

entity Process_Demo is
    port (
        a, b : in std_logic;
        y    : out std_logic
    );
end entity Process_Demo;

architecture behavioral of Process_Demo is
begin
    y <= a xor b;
end architecture behavioral;

library ieee;
use ieee.std_logic_1164.all;

entity tb_Process_Demo is
end entity tb_Process_Demo;

architecture sim of tb_Process_Demo is
    signal a, b : std_logic := '0';
    signal y : std_logic;
begin
    uut: entity work.Process_Demo
        port map (
            a => a,
            b => b,
            y => y
        );

    -- Sequential stimulus process
    process
    begin
        a <= '0'; b <= '0';
        wait for 10 ns;
        a <= '1'; b <= '0';
        wait for 10 ns;
        a <= '1'; b <= '1';
        wait for 10 ns;
        wait;
    end process;
end architecture sim;

How this Code Works (Simplified)

  • process begin ... end process - Sequential testbench block. The simulator runs each line in order, delaying at `wait for 10 ns` steps to capture output values.

7. Control Structures (If-Else & Case)

Inside a sequential process, you can model conditional routing using if-then-else and case statements.

The Train Track Analogy

Think of train tracks with a **track switcher lever**. The selection variable (like selector pin `sel`) acts as the track switcher lever. Depending on whether the lever is set to '0' or '1', the incoming train (data) is routed to Track A or Track B. In hardware synthesis, these structures compile directly into multiplexers.

If-Else Syntax

if sel = '0' then
    y <= d0;   -- Route d0 input to output
else
    y <= d1;   -- Route d1 input to output
end if;
Example: 2-to-1 Mux Simulation 
library ieee;
use ieee.std_logic_1164.all;

entity Control_Mux is
    port (
        sel, d0, d1 : in std_logic;
        y           : out std_logic
    );
end entity Control_Mux;

architecture behavioral of Control_Mux is
begin
    -- Equivalent concurrent logic path mapping selection
    y <= (d0 and not sel) or (d1 and sel);
end architecture behavioral;

library ieee;
use ieee.std_logic_1164.all;

entity tb_Control_Mux is
end entity tb_Control_Mux;

architecture sim of tb_Control_Mux is
    signal sel : std_logic := '0';
    signal d0 : std_logic := '1';
    signal d1 : std_logic := '0';
    signal y : std_logic;
begin
    uut: entity work.Control_Mux
        port map (
            sel => sel,
            d0 => d0,
            d1 => d1,
            y => y
        );

    process
    begin
        sel <= '0'; wait for 10 ns;
        sel <= '1'; wait for 10 ns;
        d0 <= '0'; d1 <= '1'; sel <= '0'; wait for 10 ns;
        sel <= '1'; wait for 10 ns;
        wait;
    end process;
end architecture sim;

How this Code Works (Simplified)

  • y <= (d0 and not sel) or (d1 and sel) - Selects between input `d0` and `d1`. When selection pin `sel` is low ('0'), output `y` connects to `d0`. When `sel` is high ('1'), `y` connects to `d1`.

8. Sequential Logic & Registers

Sequential circuits contain memory components (like registers and flip-flops) that store states and only update on clock signal transitions.

The Camera Analogy

Think of sequential logic as a **snapshot camera**:

  • The input wire `d` is the scene in front of the camera.
  • The clock pin `clk` is the **shutter button**.
  • The output `q` is the photo displayed on the camera screen.
  • The scene (`d`) can change all day, but the screen photo (`q`) remains **completely unchanged** until you press the shutter button (the rising clock edge). The moment you click it, it captures the current state of `d` and holds it until the next clock transition.
Example: D-Latch Simulation 
library ieee;
use ieee.std_logic_1164.all;

entity DLatch is
    port (
        clk, d : in std_logic;
        q      : inout std_logic
    );
end entity DLatch;

architecture behavioral of DLatch is
begin
    -- Concurrent equation modeling D-Latch behavior:
    -- q follows d when clk is high, and retains state when clk is low.
    q <= (d and clk) or (q and not clk);
end architecture behavioral;

library ieee;
use ieee.std_logic_1164.all;

entity tb_DLatch is
end entity tb_DLatch;

architecture sim of tb_DLatch is
    signal clk : std_logic := '0';
    signal d : std_logic := '0';
    signal q : std_logic;
begin
    uut: entity work.DLatch
        port map (
            clk => clk,
            d => d,
            q => q
        );

    process
    begin
        -- clk low, d high -> q remains low (retains state)
        clk <= '0'; d <= '1'; wait for 10 ns;
        
        -- clk high, d high -> q captures d and goes high
        clk <= '1'; d <= '1'; wait for 10 ns;
        
        -- clk goes low, d changes to low -> q remains high (latched!)
        clk <= '0'; d <= '0'; wait for 10 ns;
        
        -- clk goes high, d is low -> q captures d and goes low
        clk <= '1'; d <= '0'; wait for 10 ns;
        wait;
    end process;
end architecture sim;

How this Code Works (Simplified)

  • q <= (d and clk) or (q and not clk) - When clock `clk` is high ('1'), output `q` is directly wired to input `d`. When `clk` goes low ('0'), the right side `q and not clk` activates, locking in and holding the previous value of `q` in a feedback loop (memory retention).

9. Structural Modeling & Components

Structural modeling describes a circuit by declaring, instantiating, and connecting sub-modules together, similar to building on a physical breadboard.

The LEGO Blocks Analogy

Think of structural modeling as building a model with **LEGO bricks**:

  • Instead of manufacturing one custom plastic layout, you grab standard blocks (AND gates, OR gates, inverters) that are already declared (component declaration).
  • You snap them together (component instantiation) and wire their pins to create a complex design (like an XOR Gate or an Adder).
Example: XOR gate constructed structurally 
library ieee;
use ieee.std_logic_1164.all;

entity Structural_Xor is
    port (
        a, b : in std_logic;
        y    : out std_logic
    );
end entity Structural_Xor;

architecture structural of Structural_Xor is
    -- Intermediate wire declarations connecting logic blocks
    signal wire1 : std_logic;
    signal wire2 : std_logic;
begin
    -- Structural representation connecting signals
    wire1 <= a and not b;
    wire2 <= not a and b;
    y <= wire1 or wire2;
end architecture structural;

library ieee;
use ieee.std_logic_1164.all;

entity tb_Structural_Xor is
end entity tb_Structural_Xor;

architecture sim of tb_Structural_Xor is
    signal a, b : std_logic := '0';
    signal y : std_logic;
begin
    uut: entity work.Structural_Xor
        port map (
            a => a,
            b => b,
            y => y
        );

    process
    begin
        a <= '0'; b <= '0'; wait for 10 ns;
        a <= '0'; b <= '1'; wait for 10 ns;
        a <= '1'; b <= '0'; wait for 10 ns;
        a <= '1'; b <= '1'; wait for 10 ns;
        wait;
    end process;
end architecture sim;

How this Code Works (Simplified)

  • wire1 <= a and not b - Connects input `a` and inverted `b` to an AND gate, sending output to intermediate wire `wire1`.
  • y <= wire1 or wire2 - Connects `wire1` and `wire2` to an OR gate, routing the final output to pin `y`.

10. State Machines (FSM)

A Finite State Machine (FSM) is a control circuit that moves sequentially through a set of predefined states based on clock triggers and inputs.

The Traffic Light Analogy

Think of a **traffic light system**:

  • It has three distinct states: Green, Yellow, and Red.
  • It stays in Green for a fixed time, then moves to Yellow. From Yellow, it moves to Red.
  • The circuit tracks the current state, computes the target next state, and transitions on the clock edge.
Example: Toggle FSM Simulation 
library ieee;
use ieee.std_logic_1164.all;

entity Toggle_FSM is
    port (
        clk   : in std_logic;
        state : inout std_logic
    );
end entity Toggle_FSM;

architecture behavioral of Toggle_FSM is
    signal next_state : std_logic;
begin
    -- 1. Next State Decoder (State toggle logic)
    next_state <= not state;
    
    -- 2. State Register (Synchronous state memory)
    state <= (next_state and clk) or (state and not clk);
end architecture behavioral;

library ieee;
use ieee.std_logic_1164.all;

entity tb_Toggle_FSM is
end entity tb_Toggle_FSM;

architecture sim of tb_Toggle_FSM is
    signal clk : std_logic := '0';
    signal state : std_logic := '0';
begin
    uut: entity work.Toggle_FSM
        port map (
            clk => clk,
            state => state
        );

    process
    begin
        -- Clock cycle 1: Toggles state on rising clock edge
        clk <= '0'; wait for 10 ns;
        clk <= '1'; wait for 10 ns;
        
        -- Clock cycle 2: Toggles state again
        clk <= '0'; wait for 10 ns;
        clk <= '1'; wait for 10 ns;
        
        -- Clock cycle 3: Toggles state back
        clk <= '0'; wait for 10 ns;
        clk <= '1'; wait for 10 ns;
        wait;
    end process;
end architecture sim;

How this Code Works (Simplified)

  • next_state <= not state - Computes the target next state. If the light is currently off ('0'), the next state should be on ('1').
  • state <= (next_state and clk) or ... - Updates the actual register state memory wire on clock pulse high transitions.

11. Packages & Reusable Libraries

VHDL files import standard package libraries at the top of the file to gain access to reusable mathematical formulas and signal data types.

The Cookbook Analogy

Instead of manually defining how binary addition works or how standard logic levels ('0', '1', 'Z') are formatted, you import a **cookbook** (`library ieee`) and tell VHDL to use the standard recipe definitions page (`use ieee.std_logic_1164.all`). This makes code modular and saves development time.

Example: Full Adder with library imports 
library ieee;
use ieee.std_logic_1164.all;

entity Full_Adder is
    port (
        a, b, cin  : in std_logic;
        sum, cout  : out std_logic
    );
end entity Full_Adder;

architecture behavioral of Full_Adder is
begin
    -- Logical operation equations representing complete adder carry logic
    sum  <= (a xor b) xor cin;
    cout <= (a and b) or (cin and (a xor b));
end architecture behavioral;

library ieee;
use ieee.std_logic_1164.all;

entity tb_Full_Adder is
end entity tb_Full_Adder;

architecture sim of tb_Full_Adder is
    signal a, b, cin : std_logic := '0';
    signal sum, cout : std_logic;
begin
    uut: entity work.Full_Adder
        port map (
            a => a,
            b => b,
            cin => cin,
            sum => sum,
            cout => cout
        );

    process
    begin
        -- Sweep through combinations to verify Carry & Sum paths
        a <= '0'; b <= '0'; cin <= '0'; wait for 10 ns;
        a <= '1'; b <= '0'; cin <= '0'; wait for 10 ns;
        a <= '1'; b <= '1'; cin <= '0'; wait for 10 ns;
        a <= '1'; b <= '1'; cin <= '1'; wait for 10 ns;
        wait;
    end process;
end architecture sim;

How this Code Works (Simplified)

  • sum <= (a xor b) xor cin - Combines the three input bits using XOR gates to calculate the single-digit addition sum.
  • cout <= (a and b) or ... - Calculates the overflow carry bit (`cout`) so we can connect multiple adder blocks together to add large numbers.

12. Parameterization (Generics)

Generics in VHDL are used to parameterize components, allowing you to define dynamic attributes (such as bus widths or delay propagation times) without modifying the internal design logic.

The T-Shirt Factory Analogy

Imagine you run a factory that prints T-shirts. Instead of building four separate factories for Small, Medium, Large, and Extra-Large shirts, you construct one **parameterized factory**. When you start a production run, you pass in a size parameter (e.g. Width = 18 inches). Generics work exactly like this—you build one module (like a register) and pass in a parameter (width = 8 bits or 16 bits) when you instantiate it.

Example: Parameterized Buffer 
library ieee;
use ieee.std_logic_1164.all;

entity Generic_Buffer is
    generic (
        G_VAL : std_logic := '1' -- Parameterized constant
    );
    port (
        a : in std_logic;
        y : out std_logic
    );
end entity Generic_Buffer;

architecture behavioral of Generic_Buffer is
begin
    y <= a and G_VAL;
end architecture behavioral;

library ieee;
use ieee.std_logic_1164.all;

entity tb_Generic is
end entity tb_Generic;

architecture sim of tb_Generic is
    signal a : std_logic := '0';
    signal y : std_logic;
begin
    -- Instantiate with generic override map
    uut: entity work.Generic_Buffer
        generic map (
            G_VAL => '1' -- Set the generic value
        )
        port map (
            a => a,
            y => y
        );

    process
    begin
        a <= '0'; wait for 10 ns;
        a <= '1'; wait for 10 ns;
        wait;
    end process;
end architecture sim;

How this Code Works (Simplified)

  • generic ( G_VAL : std_logic := '1' ) - Defines a parameter called `G_VAL` with a default state of high ('1').
  • generic map ( G_VAL => '1' ) - The testbench instantiates the component and explicitly overrides the parameter `G_VAL` with the desired value.

13. Functions & Procedures

VHDL subprograms (Functions and Procedures) are behavioral blocks used to group common calculations, keeping your design clean and reusable.

The Kitchen Blender Analogy

Instead of manually chopping, mixing, and pouring ingredients every time you want a smoothie, you buy a blender (a **Function**). You put in ingredients (inputs) and press a single button to get a smoothie back (the return output). In VHDL, functions perform instant math/logic and return a single value, while procedures can perform multiple actions and modify multiple output signals.

Example: Function-based Logic Gate 
library ieee;
use ieee.std_logic_1164.all;

entity Function_Demo is
    port (
        a, b : in std_logic;
        y    : out std_logic
    );
end entity Function_Demo;

architecture behavioral of Function_Demo is
    -- Helper function to perform XOR operation
    function my_xor (x, y : std_logic) return std_logic is
    begin
        return x xor y;
    end function my_xor;
begin
    -- Calling the function
    y <= my_xor(a, b);
end architecture behavioral;

library ieee;
use ieee.std_logic_1164.all;

entity tb_Function is
end entity tb_Function;

architecture sim of tb_Function is
    signal a, b : std_logic := '0';
    signal y : std_logic;
begin
    uut: entity work.Function_Demo
        port map (a => a, b => b, y => y);
    process
    begin
        a <= '0'; b <= '0'; wait for 10 ns;
        a <= '1'; b <= '0'; wait for 10 ns;
        a <= '1'; b <= '1'; wait for 10 ns;
        wait;
    end process;

How this Code Works (Simplified)

  • function my_xor (...) return std_logic - Declares a reusable logic box. Whenever we call `my_xor`, it takes the two inputs, performs the XOR math, and outputs the result.
  • y <= my_xor(a, b) - Calls our custom blender function, wiring the inputs `a` and `b` and routing the result directly to pin `y`.

14. Self-Checking Testbenches & Asserts

In professional design, manually checking waveforms is slow and error-prone. We write self-checking testbenches using `assert` statements to automate verification.

The Smoke Detector Analogy

Instead of sitting in front of your oven for hours watching your food cook, you install a **smoke detector** (an **Assert** statement). If smoke occurs, the alarm rings (reports an error). In VHDL, `assert` checks if a condition is true. If it is false, the simulator raises an alarm, printing your custom message and stopping execution.

Example: Assert-based Testbench Verification 
library ieee;
use ieee.std_logic_1164.all;

entity Gate_Checked is
    port (
        a : in std_logic;
        y : out std_logic
    );
end entity Gate_Checked;

architecture behavioral of Gate_Checked is
begin
    y <= not a;
end architecture behavioral;

library ieee;
use ieee.std_logic_1164.all;

entity tb_Assert_Demo is
end entity tb_Assert_Demo;

architecture sim of tb_Assert_Demo is
    signal a : std_logic := '0';
    signal y : std_logic;
begin
    uut: entity work.Gate_Checked
        port map (
            a => a,
            y => y
        );

    process
    begin
        a <= '0'; wait for 10 ns;
        -- Verify that y is indeed '1' when a is '0'
        assert (y = '1')
            report "ERROR: Output y should be 1 when a is 0"
            severity error;

        a <= '1'; wait for 10 ns;
        -- Verify that y is indeed '0' when a is '1'
        assert (y = '0')
            report "ERROR: Output y should be 0 when a is 1"
            severity error;
        wait;
    end process;
end architecture sim;

How this Code Works (Simplified)

  • assert (y = '1') report ... severity error; - Tells the simulator to verify that output wire `y` has a voltage of '1'. If it does not, the simulator displays the custom message and marks the simulation run as failed.

15. Custom Arrays & Records

VHDL allows you to create custom composite data types, grouping related wires under a single signal name using **Arrays** and **Records**.

Spreadsheets vs Contact Cards Analogy

  • Arrays: Like a **numbered column** in an Excel sheet. Every cell in that column must store the exact same type of data (e.g. 16 sensors each storing a bit). In VHDL, we use arrays to model hardware memory arrays (RAM and ROM).
  • Records: Like a **business contact card**. It contains different types of fields grouped together (Name = string, Age = integer, Active = boolean). In VHDL, records are used to route packets containing control lines and data lines together.

16. Synthesis vs Simulation

A crucial VHDL concept is understanding the difference between code meant to run inside a software simulator and code compiled into physical copper gates on an FPGA/ASIC chip.

The Blueprint vs Video Game Analogy

  • Simulation: Like playing a **virtual video game**. In a game, you can teleport, fly, or freeze gravity. In VHDL simulation, you can use time delays (`wait for 10 ns;`) and read files from your computer because everything runs virtually.
  • Synthesis: Like building a **real brick house**. You must follow physical rules. You cannot teleport materials. Synthesizable VHDL is code that the compiler can translate into physical copper transistors, wires, and gates.

Golden Rules of Synthesizable VHDL

  • Never use "wait for" delays: Real gates cannot delay electricity by exactly 10 ns dynamically. We use clock pulses (`rising_edge(clk)`) to coordinate time steps.
  • Use Resets for Initial States: While `signal x : std_logic := '0';` works in simulator memory, physical transistors power up in random states. You must use a hardware reset line to force registers into a clean state.
  • Close all code loops: Ensure every `if` statement has an `else` branch, and every `case` statement covers `when others`. This prevents the compiler from creating accidental memory loops called "latches".

By keeping simulation-only constructs inside testbenches and synthesizable structures inside your design entities, your VHDL code will deploy successfully onto real FPGA silicon!

17. Delay Timing Models (Inertial vs Transport)

VHDL provides timing models to simulate the physical transmission times of electrical charges flowing through gates and traces.

Inertial Delay vs Transport Delay

  • Inertial Delay (Default): Like a **furnace heating system**. If you click a heater switch on and off in one millisecond, the heater does not start because it requires continuous energy to heat up. Inertial delay filters out narrow transient pulses (called spikes/glitches). If a pulse width is shorter than the delay time, the signal is rejected.
  • Transport Delay: Like a **water hose**. Water enters one end of the hose and travels to the other. Every bubble, pressure wave, and drop is delivered to the exit, just delayed in time. Transport delay represents ideal wire delays that transmit all glitches.
Example: Timing Delay Declarations 
library ieee;
use ieee.std_logic_1164.all;

entity Delay_Demo is
    port (
        x : in std_logic;
        y_inertial  : out std_logic;
        y_transport : out std_logic
    );
end entity Delay_Demo;

architecture behavioral of Delay_Demo is
begin
    -- 1. Inertial Delay: Rejects pulses narrower than 5ns
    y_inertial <= x after 5 ns;

    -- 2. Transport Delay: Transmits all pulses, simply shifted
    y_transport <= transport x after 5 ns;
end architecture behavioral;

library ieee;
use ieee.std_logic_1164.all;

entity tb_Delay_Demo is
end entity tb_Delay_Demo;

architecture sim of tb_Delay_Demo is
    signal x : std_logic := '0';
    signal y_inertial, y_transport : std_logic;
begin
    uut: entity work.Delay_Demo
        port map (
            x => x,
            y_inertial => y_inertial,
            y_transport => y_transport
        );

    process
    begin
        x <= '0'; wait for 10 ns;
        x <= '1'; wait for 10 ns;
        x <= '0'; wait for 2 ns;  -- Spike pulse of 2ns (narrower than 5ns)
        x <= '1'; wait for 10 ns;
        wait;
    end process;
end architecture sim;

How this Code Works (Simplified)

  • y_inertial <= x after 5 ns - Electrically delays `x` by 5 nanoseconds, filtering out spikes shorter than 5 ns.
  • y_transport <= transport x after 5 ns - Ideal wire path. Shifts the input waveform of `x` by 5 ns without filtering out glitches.

18. Signal Attributes

Attributes in VHDL are built-in functions that query information about signals, arrays, and variables (such as checking clock edges or sizing arrays).

Common Attributes

  • s'event - Returns true if a voltage change just occurred on signal `s`. (Crucial for clock detection: `clk'event and clk='1'`).
  • s'last_value - Queries the previous value of signal `s` prior to the latest transaction event.
  • array'length - Returns the number of items in a bus array.
  • type'image(value) - Converts a value to a readable text string (e.g. `std_logic'image(q)`).
Example: Attribute Event Monitor 
library ieee;
use ieee.std_logic_1164.all;

entity Attribute_Monitor is
    port (
        clk : in std_logic;
        q   : out std_logic
    );
end entity Attribute_Monitor;

architecture behavioral of Attribute_Monitor is
begin
    -- q follows clk
    q <= clk;
end architecture behavioral;

library ieee;
use ieee.std_logic_1164.all;

entity tb_Attribute_Monitor is
end entity tb_Attribute_Monitor;

architecture sim of tb_Attribute_Monitor is
    signal clk : std_logic := '0';
    signal q   : std_logic;
begin
    uut: entity work.Attribute_Monitor
        port map (
            clk => clk,
            q => q
        );

    process
    begin
        clk <= '0'; wait for 10 ns;
        clk <= '1'; wait for 10 ns;

        report "Signal q value converted to text: " & std_logic'image(q);

    end process;

How this Code Works (Simplified)

  • std_logic'image(q) - Converts the current signal state ('0', '1', 'Z') of `q` into a string so it can be printed to the terminal console during simulation.

19. Multiple Drivers & Resolution Functions

In digital electronics, connecting multiple output pins directly together to drive the same copper wire creates a conflict. VHDL uses resolution functions to calculate the net voltage.

The Tug of War Analogy

Imagine two children pulling a rope in opposite directions: one pulls left (drives wire to '0'), the other pulls right (drives wire to '1'). The rope represents a **multi-driven wire**. VHDL's resolved type `std_logic` uses a standard 9-state resolution table. A strong driver ('0' or '1') overrides a weak driver (high impedance 'Z'). If two strong drivers oppose, the state becomes unknown ('X').

Example: Tri-state Driver Simulation 
library ieee;
use ieee.std_logic_1164.all;

entity Tristate_Bus is
    port (
        enable : in std_logic;
        data   : in std_logic;
        bus_line : out std_logic
    );
end entity Tristate_Bus;

architecture behavioral of Tristate_Bus is
begin
    -- If enabled, drive data; else release the line (High Impedance 'Z')
    bus_line <= data when enable = '1' else 'Z';
end architecture behavioral;

library ieee;
use ieee.std_logic_1164.all;

entity tb_Tristate is
end entity tb_Tristate;

architecture sim of tb_Tristate is
    signal enable : std_logic := '0';
    signal data   : std_logic := '1';
    signal bus_line : std_logic;
begin
    uut: entity work.Tristate_Bus
        port map (
            enable => enable,
            data => data,
            bus_line => bus_line
        );

    process
    begin
        -- Bus is released ('Z')
        enable <= '0'; wait for 10 ns;
        
        -- Bus is active ('1')
        enable <= '1'; wait for 10 ns;
        wait;
    end process;
end architecture sim;

How this Code Works (Simplified)

  • bus_line <= data when enable = '1' else 'Z' - This acts as a tri-state switch. When `enable` is low, the connection is broken ('Z' high impedance state), freeing the bus line so other chips can use it.

20. Block Statements

A block statement is a concurrent structure used to partition a large, messy architecture into distinct, localized sections.

The Room Partition Analogy

Imagine a large open-plan office. To separate different departments, you install temporary room dividers (a **Block**). The partitions do not change the work being done, but they group related desks and local tools (signals) together to prevent clutter. Blocks can also contain guard clauses that only enable local assignments under specific conditions.

Example: Block Partition Simulation 
library ieee;

architecture behavioral of Block_Demo is
begin
    -- Partition block grouping logic
    adder_block : block
        signal local_wire : std_logic;
    begin
        local_wire <= a and b;
        y <= local_wire;
    end block adder_block;
end architecture behavioral;

library ieee;
use ieee.std_logic_1164.all;

entity tb_Block_Demo is
end entity tb_Block_Demo;

architecture sim of tb_Block_Demo is

    signal y    : std_logic;
begin
    uut: entity work.Block_Demo
        port map (
            a => a,
            b => b,
            y => y
        );

        a <= '0'; b <= '1'; wait for 10 ns;

    end process;

  • adder_block : block ... end block - Declares a partition boundary. The wire `local_wire` exists only inside this block and cannot be accessed by other blocks.

21. VHDL Configurations

Configurations in VHDL bind specific architectures to specific entities in a design hierarchy, similar to swapping plugins or skins in software.

The Phone Case Analogy

Think of your smartphone. The smartphone interface (Entity buttons, screen ports) remains identical, but you can snap on different phone cases (Architectures: e.g. a silicon case, a leather case, or a battery case). In VHDL, you can build a fast, abstract simulation model and a slow, detailed gates model for the same entity, then use a `configuration` file to select which model to test.

23. VHDL Coding Style Standards

VHDL code is meant to be read by humans and synthesized by compilers. Following standard coding style guides prevents bugs and makes collaboration easier.

The Blueprint Formatting Analogy

If an architect draws a blueprint, they follow standard symbols: blue lines for pipes, yellow lines for gas, and green lines for electricity. If they drew them arbitrarily, construction workers would connect wires to pipes. VHDL style guides act as these standard rules, keeping designs clear and consistent.

Core Style Standards for VHDL

  • Suffix Mappings: Use consistent naming suffixes for different objects:
    • _in / _out: For Entity interface input and output pins (e.g. `clk_in`, `data_out`).
    • _sig / _w: For internal signals and wire connections (e.g. `clk_sig`, `enable_w`).
    • _tb: For testbench files.
  • Indentation: Always indent code inside entities, architectures, and process blocks by 4 spaces.
  • Type Safety: Never mix `std_logic_vector` with `integer` mathematical operations without importing `numeric_std` and performing explicit type conversion: 
    -- Correct numeric conversion:
sig_sum <= std_logic_vector(unsigned(bus_a) + unsigned(bus_b));
  • Single-purpose Processes: Do not mix combinational logic and sequential memory logic inside the same process block. Write separate processes for state registers and state decoders.

By following these VHDL formatting guidelines, your designs will be readable, maintainable, and synthetically correct!
